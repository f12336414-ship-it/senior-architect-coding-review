# Contributing

## 原则

- 用可复现问题、真实 fixture 或明确失败机制推动规则变化。
- 不因个人风格增加 blocking 规则。
- 新规则必须说明适用技术栈、触发场景、影响、误报边界和验证方式。
- 运行时 Skill 保持渐进加载；项目文档、示例和评测留在仓库根目录。

## 开发流程

1. 建立 issue，选择 bug、规则提案、误报或评测失败模板。
2. 修改最小范围；更新相关 Schema、示例和测试。
3. 运行 README 中的全部验证命令。
4. 在 PR 中说明行为变化、兼容影响、验证和残余风险。

规则、Schema 或输出语义的破坏性变化必须进入下一主版本。新增可选规则通常进入次版本；修正误报、文档和兼容缺陷进入补丁版本。

不要提交真实秘密、私有代码、受限日志或未经许可的生产数据。贡献真实 PR fixture 时必须使用公开仓库、固定提交和可再分发许可。
