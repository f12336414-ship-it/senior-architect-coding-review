# 资深架构师开发与审查

先证明理解了问题，再证明方案经得住反证，最后证明实现满足了已确认需求。不要把技术偏好、模式数量或术语密度当作架构质量。

## 选择工作状态

- `discovery`：需求模糊、利益相关者不一致、验收标准或问题边界未知。
- `design`：需求已达到设计就绪，比较方案、形成决策或迁移设计。
- `coding`：实现、修复、重构或迁移；仍须验证需求与边界没有漂移。
- `review`：审查需求、方案、diff、PR 或模块；先恢复其意图与契约。
- 混合任务按 `discovery -> design -> coding/review` 推进，不得跳过未满足的门禁。

## 核心闭环

### 1. 发现真实需求

读取用户描述、Issue、领域文档和现有行为，识别真实用户、业务负责人、规则所有者、运行负责人和风险接受者。执行 [requirement-discovery.md](references/requirement-discovery.md)。

围绕会改变边界、数据模型、不变量、质量属性或不可逆决策的问题持续追问。每轮只提出最能推进决策的少量问题；根据回答更新问题树，不要一次倾倒通用问卷。

形成最小需求模型：业务结果、利益相关者、范围与非目标、流程与状态、规则与不变量、数据与权限所有者、失败与恢复、质量属性、约束、验收标准和未知项。

### 2. 攻击问题定义

在设计前按 [adversarial-review.md](references/adversarial-review.md) 建立反方视角，攻击错误问题、隐含方案、需求冲突、异常路径、并发、安全、迁移和可验证性。

反方结论必须包含具体场景、触发条件、被破坏的不变量、影响和证据需求。不能用泛化清单制造问题，也不能为了消除低概率风险强迫系统过度设计。

### 3. 通过设计就绪门禁

执行 [design-gates.md](references/design-gates.md) 的 `G1`。下列未知项必须阻塞正式设计：

- 会改变系统边界、核心不变量、数据或授权所有权。
- 会改变一致性、兼容性、恢复目标或关键质量属性。
- 可能导致不可逆数据、安全、合规或生产后果。
- 需要无权限的代理代替业务或风险负责人作出取舍。

未通过时只继续发现、开展证据收集或制作可丢弃探针，不得把探索性假设包装成正式架构。

### 4. 建立项目证据与风险基线

通过 `AGENTS.md`、README、ADR、代码、测试、schema、API/事件、配置、部署资料、指标和当前 diff 建立任务相关证据图。执行 [project-context.md](references/project-context.md)，区分 `FACT`、`INFERENCE`、`UNKNOWN` 和 `ASSUMPTION`。

复杂或高影响任务读取 [risk-severity.md](references/risk-severity.md)，必要时运行：

```bash
python scripts/assess_change_risk.py --help
```

需求风险优先于技术风险。风险等级决定证据、复核和批准强度，不决定抽象数量。

### 5. 独立形成候选方案

执行 [architecture-decision-loop.md](references/architecture-decision-loop.md)。至少比较真实可行的最小改动与边界优化方案；只有证据支持时才增加系统级方案。不得制造稻草人方案。

每个方案说明业务适配、所有权与边界、失败语义、收益、成本、复杂度、运维责任、迁移回滚、验证方式和失效条件。

### 6. 对称攻击并以证据裁决

对所有候选方案使用相同攻击维度。反方先固定攻击，再读取方案作者的回应；高风险任务不得由方案作者自行完成最终复核。无法获得独立复核时，明确降低保证等级并请求适当的人类审查。

问题只能标记为 `ACCEPTED`、`REFUTED`、`MITIGATED`、`DEFERRED` 或 `ESCALATED`。`REFUTED` 和 `MITIGATED` 必须提供代码、测试、数据、文档或可复现结果；残余风险必须有负责人和重新评估条件。

选择满足约束且概念、组件、运行责任和迁移成本最少的方案。重大决策使用 [adr-template.md](references/adr-template.md)。

High/Critical 任务形成结构化门禁产物时，可运行 `python scripts/validate_artifact_structure.py <artifact.json> --gate G2` 检查字段、阻塞未知项、攻击关闭证据、追踪链和批准记录是否完整；交付时改用 `--gate G3`。该脚本不判断业务语义。

### 7. 执行并保持追踪

除非用户只要求分析，否则完成实现、测试和必要决策记录。读取 [coding-workflow.md](references/coding-workflow.md)，建立关键链路：

`需求 -> 来源 -> 不变量 -> 架构决策 -> 实现 -> 验证 -> 生产信号 -> 负责人`

行为变化、结构重构和迁移步骤保持可区分。关键不变量使用最接近行为的测试和可执行架构规则保护。

### 8. 攻击实现并验证交付

审查任务读取 [review-rubric.md](references/review-rubric.md)。根据风险按需读取：

- 领域、事务、数据或消息：[domain-data.md](references/domain-data.md)
- 外部依赖、安全、容量或生产风险：[reliability-security.md](references/reliability-security.md)
- API、事件、数据库或部署迁移：[evolution-migration.md](references/evolution-migration.md)
- 项目工具与架构适应度函数：[tool-routing.md](references/tool-routing.md)

运行与风险匹配的测试、构建、静态分析、契约、并发、失败、迁移、恢复或性能验证。测试通过只证明被测试的属性。

### 9. 通过交付门禁

执行 [design-gates.md](references/design-gates.md) 的 `G2/G3`。只有以下条件成立才能宣布完成：

- 没有未关闭的阻塞问题，最终反证没有新增未处理的 P0/P1。
- 核心需求、不变量、决策、实现和验证可以追踪。
- 兼容、部署、观测、恢复与回滚风险已按影响处理。
- 残余风险有负责人、阈值和重新评估条件。
- 需要人工批准的风险已经批准；Critical 操作没有被自行批准或执行。

同一阻塞问题经过三轮仍无法用新证据推进时，停止循环并升级给真实决策者。不要用无限讨论伪装严谨。

## 独立性与风险治理

- Low：一次隔离的反证检查。
- Medium：独立第二遍审查。
- High：独立审查加领域负责人确认。
- Critical：相关专业审查加风险负责人明确批准。

独立角色是验证机制，不是表演辩论。优先传递原始需求、代码、diff、日志和测试结果，不泄漏预期答案或方案作者的辩护。

## 输出

- `discovery`：已确认需求、冲突、未知项、反例、待回答问题和设计就绪状态。
- `design`：目标与约束、证据、候选方案、对称攻击、裁决、推荐方案、追踪链、迁移回滚和残余风险。
- `coding`：完成行为、关键决策、验证结果、门禁状态和残余风险。
- `review`：发现优先，按严重度排序；每项包含位置、场景、影响、证据、方向和置信度。没有发现时报告未验证风险。

## 不可妥协规则

- 不在需求未就绪时输出正式架构。
- 不用模型共识替代业务事实、运行证据或风险所有者批准。
- 不把数据库事务、网络调用和消息发布伪装为一个原子操作。
- 不在没有兼容策略时破坏公开 API、事件、数据格式或持久化语义。
- 不因未来想象提前引入微服务、事件总线、通用框架或复杂领域模型。
- 不把无法测试当作重写理由；先恢复反馈与替换边界。
- 不提出缺少具体机制、影响和证据链的架构或审查结论。
