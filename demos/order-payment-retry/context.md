# Demo 项目上下文

## 已知事实

- 订单服务负责创建订单和保存本地支付状态。
- 外部支付提供方负责实际扣款。
- 当前系统把 HTTP 超时记录为 `failed`。
- 当前没有稳定的 `payment_intent_id`，只保存订单号和最近一次 provider 响应摘要。
- 客服可以人工退款，但没有自动重复扣款检测。
- 账务每日对账，不能实时阻止重复扣款。

## 未知项

- provider 超时后是否可能已经扣款成功。
- provider 是否支持幂等键，以及幂等键的生命周期和冲突语义。
- 哪些失败码可安全重试，哪些必须人工处理。
- 最大重试次数、重试窗口、退避策略和用户通知规则。
- 重复扣款的风险接受者和赔付规则。
- 订单状态、支付状态和账务状态之间的权威关系。

## 风险边界

- 支付、账务和退款属于高风险路径。
- 任何自动扣款逻辑必须有领域负责人和风险负责人批准。
- 没有幂等协议、对账和停止阈值时，Agent 不应自行实现自动重试。
