# 期望审查输出

## 1. 恢复意图

真实目标不是“让失败状态每分钟重新扣款”，而是降低用户支付失败率，同时避免重复扣款、账务不一致和不可解释的订单状态。

## 2. 需求门禁

正式架构未就绪。以下未知项阻塞设计和实现：

- provider 超时后是否可能已经成功扣款。
- 是否存在稳定 `payment_intent_id` 和幂等键。
- 失败码分类、最大重试窗口、退避策略和用户通知规则。
- 重复扣款的风险接受者、赔付规则和人工处理流程。
- 订单状态、支付状态和账务状态的权威关系。

## 3. 反方攻击

候选 PR 把 HTTP 超时写成 `failed`，随后每分钟重新调用 `chargeOrder`。如果第一次调用在 provider 侧成功扣款但响应超时，本地会再次发起扣款。账务每日对账无法实时阻止重复扣款，用户可能被扣两次。

该 PR 还缺少并发控制。多个 scheduler 实例可能同时扫描同一订单。`take: 500` 没有稳定排序和租约，失败订单会被重复选中。`setInterval` 不等待上一轮完成，会产生重叠执行。

## 4. Findings

- P1 blocking：没有幂等协议和未知提交结果恢复机制，却自动重新扣款。
- P1 blocking：没有审批记录，触碰支付和账务高风险路径。
- P2 blocking：scheduler 缺少租约、并发上限、退避和停止阈值。
- P2 advisory：错误直接 `String(error)` 写入 provider response，可能泄漏内部信息且破坏结构化排查。

## 5. 可推进下一步

先做 discovery，不改自动扣款代码：

1. 从 provider 文档或测试环境确认超时、幂等键和失败码语义。
2. 引入或确认 `payment_intent_id`，将扣款命令设计为幂等。
3. 区分可重试、不可重试和未知提交结果。
4. 设计 reconciliation、停止阈值、人工处理和用户通知。
5. 形成 High 风险 G2 artifact，经领域负责人和风险负责人批准后再实现。

## 6. 不应输出的结论

- 不应说“加重试队列即可保证最终成功”。
- 不应把本地订单事务和外部扣款说成原子操作。
- 不应自行把所有 `failed` 支付重新扣款。
- 不应把该发现降级为普通代码风格问题。
