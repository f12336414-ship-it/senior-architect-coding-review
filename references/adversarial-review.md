# 独立反证协议

## 目的

通过具体反例发现错误的问题定义、方案盲点和实现缺陷。反方的目标不是否定一切，也不是增加组件，而是寻找能够推翻当前结论的最小真实场景。

## 独立性

- 让方案作者、反方和证据裁决成为不同逻辑角色。
- 先向反方提供原始需求、项目事实和待审对象，不提供预期答案或作者辩护。
- 让反方固定第一轮攻击后，再允许作者回应。
- 让裁决者重新读取原始证据，不以双方措辞或信心作为依据。
- High/Critical 任务优先使用独立人员、独立代理或专业审查；做不到时明确保证等级下降。

同一模型切换角色只能算隔离复核，不能宣称真正独立。

## 三次攻击

### 1. 攻击问题定义

- 用户目标是否被某个预设方案替代？
- 利益相关者、规则所有者或风险接受者是否缺席？
- 正常、拒绝、取消、超时、恢复和人工路径是否完整？
- 规则、状态、数据所有权和验收标准是否矛盾？
- 验收指标能否被错误实现或局部优化轻易“刷过”？
- 需求是否依赖未确认的规模、合规、组织或生产假设？

### 2. 对称攻击候选方案

对所有候选使用相同且与风险相关的维度：

- 业务适配、所有权、内聚与依赖方向。
- 事务、并发、幂等、重复、乱序和部分成功。
- 身份、授权、租户、秘密、输入和供应链。
- 超时、重试、过载、背压、隔离、降级和恢复。
- API、事件、schema、配置、部署顺序和历史数据。
- 尾延迟、容量、成本和不可水平扩展资源。
- 团队技能、运维负担、认知复杂度和长期演进。

先寻找能够淘汰方案的条件，再讨论优化。禁止制造稻草人，也禁止只攻击未被偏好的方案。

### 3. 攻击实现与运行

- 从入口沿状态变化、持久化、外部调用和返回路径追踪。
- 构造边界值、并发、重复、失败、取消、恢复和恶意输入。
- 检查实现是否偏离需求、不变量、ADR 或迁移顺序。
- 检查测试证明了什么以及没有证明什么。
- 检查日志、指标和告警能否发现真实用户影响。

## 攻击格式

```text
ID：ATK-n
标题：
对象：需求 | 方案 | 实现
场景：具体输入、并发、故障或攻击步骤
触发条件：
被破坏的不变量或承诺：
影响：
证据：当前代码、测试、数据或明确推理链
严重度：P0 | P1 | P2 | P3
需要的裁决证据：
```

缺少具体可达场景和影响的问题不得列为阻塞缺陷。

## 回应与裁决

只允许以下状态：

- `ACCEPTED`：问题成立，修改需求、方案或实现。
- `REFUTED`：用可检查证据证明场景不可达或结论错误。
- `MITIGATED`：增加控制并验证风险已降低。
- `DEFERRED`：不影响当前决策，指定负责人、期限或触发条件。
- `ESCALATED`：需要领域、业务、安全、运维或风险负责人裁决。

`REFUTED` 和 `MITIGATED` 必须引用证据。`DEFERRED` 不能用于 P0；P1 延期必须由有权限的人明确接受。

## 复杂度反证

反方提出的每项控制同样接受审查：

```text
风险降低收益是否大于
新增组件 + 认知负担 + 运维负担 + 故障面 + 迁移成本？
```

优先使用约束、测试、限流、可逆发布和现有平台能力。只有真实风险无法用更小机制控制时才增加结构。

## 退出条件

- 所有 P0/P1 均已接受修正、证据反驳、有效缓解或授权升级。
- 最终隔离复核没有产生新的未处理 P0/P1。
- P2/P3 不会阻止当前目标，并有明确处置。
- 残余风险、负责人和重新评估条件已记录。

同一问题经过三轮没有新证据时停止角色辩论，升级给真实决策者。
