# 需求发现与问题建模

## 目标

在设计技术方案前，确认真正要解决的业务问题以及足以改变架构决策的约束。不要追求知道所有细节；追问到所有决策性不确定性已解决、被安全隔离或交由有权限的人裁决。

## 发现顺序

### 1. 确认责任关系

- 谁提出需求，谁是真实用户，谁会受到副作用影响？
- 谁拥有业务规则、权威数据和验收决定？
- 谁运行系统、响应事故并承担新增运维责任？
- 谁有权限接受成本、合规、数据或可用性残余风险？

同一个人可以承担多个角色，但不得把角色缺失默认为代理拥有决策权。

### 2. 描述业务结果

使用可观察结果描述目标，不要从技术方案开始：

```text
当 <参与者/事件> 在 <业务情境> 下发生时，
系统需要产生 <可观察结果>，
以改善 <业务指标或风险>，
同时不得破坏 <现有承诺>。
```

追问当前流程、痛点、频率、影响对象、成功指标、时间约束和不做什么。区分用户说出的方案与其背后的需要。

### 3. 建立行为与状态模型

- 列出正常流程、拒绝、取消、超时、恢复和人工处理。
- 识别状态、合法转换、终态以及谁可以触发转换。
- 明确重复请求、并发操作、乱序事件和部分成功的语义。
- 区分必须保留的现有行为、历史偶然行为和待修复缺陷。

### 4. 提取不变量与所有权

每条关键规则记录：

```text
ID：INV-n
规则：始终必须成立的业务事实
所有者：有权定义该规则的角色或模块
数据：证明该规则所需的权威数据
边界：必须原子成立还是允许延迟
失败：违反后对用户、数据或合规的影响
```

### 5. 定义质量属性场景

不要只写“高性能”“高可用”。使用可验证场景：

```text
刺激：发生什么
环境：正常、峰值、故障或恢复期
对象：受影响的服务、数据或用户
响应：系统应如何处理
度量：延迟、吞吐、错误预算、RPO/RTO 或容量阈值
```

至少考虑安全、隐私、一致性、兼容性、性能、可用性、恢复、成本和可运维性中与当前任务有关的部分。

## 自适应追问

每轮只询问最可能改变边界、数据、不变量、质量属性或不可逆决策的 1–3 个问题。收到回答后：

1. 更新已确认事实。
2. 找出新术语、矛盾和隐含假设。
3. 生成下一层问题。
4. 判断能否从仓库、指标或现有文档获得答案。
5. 只有需要真实决策时才继续询问用户。

不要重复询问可从项目中发现的信息，也不要为了完整表格阻塞可逆的局部工作。

## 最小需求模型

```text
业务结果：
利益相关者与职责：
范围：
非目标：
当前与目标流程：
状态与转换：
规则与不变量：
数据和授权所有者：
失败、恢复与人工处理：
质量属性场景：
兼容与迁移约束：
验收标准：
未知项及其决策影响：
```

对于局部低风险改动，可以在工作记录中维护该模型；只有跨边界或高风险任务才需要形成正式文档。

## 停止追问条件

停止需求发现需要同时满足：

- 能用业务语言说明要改变什么结果以及为何改变。
- 能描述正常、失败和恢复路径。
- 核心不变量、数据与规则所有者明确。
- 关键质量属性拥有可验证阈值或明确取舍者。
- 范围、非目标和验收标准可以区分成功与失败。
- 剩余未知不会改变核心方案，或已有可逆隔离和负责人。

停止追问不等于需求永不变化。记录会重新打开需求发现的信号。
