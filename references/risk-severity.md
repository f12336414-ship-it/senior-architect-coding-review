# 风险治理与控制强度

风险用于决定需求澄清、证据、复核和批准强度。审查发现严重度使用 [review-rubric.md](review-rubric.md)，不要混淆二者。

## 四类风险

### 需求风险

- 目标、验收、范围或非目标不清楚。
- 规则所有者、风险接受者或利益相关者存在冲突。
- 核心不变量、数据所有权或质量属性依赖未知假设。

需求风险可以阻止任何技术风险等级的任务。代码改动很小不代表实现错误需求的风险很小。

### 方案风险

- 影响范围、数据、授权、兼容性和可用性。
- 事务、进程、部署或信任边界变化。
- 回滚困难、不可逆、可观测性弱或技术新颖。

### 执行风险

- 测试、环境、发布权限或恢复能力不足。
- 多版本并存、迁移顺序、容量或依赖行为无法验证。

### 治理风险

- 缺少领域、安全、运维或合规负责人。
- 重大残余风险没有授权接受者。
- 方案作者同时充当唯一审查和批准者。

总风险取各类风险的最高等级，并应用不可逆、安全、数据和生产组合升级规则；不要用低风险项平均掉高风险项。

## 风险下限

| 条件 | 最低等级 |
| --- | --- |
| 部分明确的决策性需求 | Medium |
| 冲突或未知的决策性需求 | High |
| 敏感数据 | Medium |
| 身份、授权、租户边界 | High |
| 监管边界 | Critical |
| 生产数据迁移 | Medium |
| 不可逆数据变化 | Critical |
| 公共 API 或消费者契约 | Medium |
| 协议级变化 | High |
| 已知破坏性生产变更 | Critical |
| 多系统影响 | High |
| 困难回滚 | High |
| 不可逆操作 | Critical |

可使用脚本获得一致的初步结果：

```bash
python scripts/assess_change_risk.py \
  --environment production \
  --requirements clear \
  --blast-radius service \
  --data migration \
  --security sensitive \
  --compatibility public \
  --availability degraded \
  --rollback coordinated \
  --observability partial \
  --novelty some
```

未提供的输入默认为 `unknown` 并提高风险，而不是默认安全。脚本输出是最低控制要求，不替代项目事实和授权人的判断。

## 控制等级

### Low

- 确认局部目标和验收标准。
- 运行相关单元测试与静态检查。
- 执行一次隔离反证，保持直接回退能力。

### Medium

- 形成简短需求与约束记录。
- 增加集成、契约或失败路径验证。
- 使用独立第二遍审查。
- 确认指标、日志、兼容和回滚步骤。

### High

- 通过正式设计就绪和实现就绪门禁。
- 形成书面设计或 ADR、追踪链和发布顺序。
- 使用独立审查并获得领域负责人确认。
- 分阶段发布并演练失败、恢复、回滚和数据校验。
- 设置成功、停止和事故升级阈值。

### Critical

- 不允许代理自行批准或执行不可逆步骤。
- 使用相关领域、安全、数据或运维专业审查。
- 拆分为可逆阶段和人工停止点。
- 在真实规模副本上演练迁移、恢复和前向修复。
- 获取风险负责人明确批准并保留证据。

## 风险降低证据

成熟模式、强契约测试、可执行架构规则、可观测性、可逆发布和已演练恢复可以降低执行风险，但不能自动消除需求冲突、监管要求或不可逆后果。
