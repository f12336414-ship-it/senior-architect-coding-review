# 架构决策闭环

## 1. 描述决策

用一句话描述必须作出的决策，而不是描述想使用的技术。

错误示例：是否使用事件驱动架构？

正确示例：订单确认后，如何在不扩大结账事务的情况下可靠触发履约？

## 2. 提取约束

至少确认：

- 业务目标、时间约束和明确非目标。
- 一致性、延迟、吞吐、可用性和恢复要求。
- 数据、隐私、安全和合规要求。
- 团队技能、运维能力、预算和现有技术约束。
- 兼容窗口、迁移顺序和回滚能力。

不要把偏好写成约束。

## 3. 建立候选方案

至少比较：

- 保持当前结构的最小改动。
- 修正局部边界或职责的中等改动。
- 只有证据支持时才考虑系统级变化。

每个方案说明收益、成本、风险、可逆性、运维负担和失效条件。不要制造明显劣质方案衬托预选答案。

## 4. 检查关键维度

- **所有权**：规则、数据和决策是否有唯一权威位置？
- **耦合**：修改是否扩散到无关模块？耦合是语义、时间、部署还是数据耦合？
- **内聚**：共同变化的行为是否放在一起？
- **依赖**：稳定业务策略是否依赖易变框架或基础设施？
- **复杂度**：新概念是否减少整体认知负担，而非只移动复杂度？
- **运行时**：失败、过载、重试和部分成功如何传播？
- **演进**：方案是否支持兼容迁移、观测和删除旧路径？

## 5. 选择最小充分方案

优先选择：

- 满足当前约束且概念最少的方案。
- 失败边界明确、可测试、可观测和可回滚的方案。
- 对真实变化轴提供扩展点，而非对想象中的未来泛化。

以下情况才引入新抽象：存在第二个真实实现、需要隔离外部系统、需要保护核心策略，或重复知识已开始分叉。

以下情况才建议拆分服务：边界拥有独立业务能力和数据，确有独立扩缩容、隔离、合规或交付需求，并能承担分布式成本。

## 6. 记录重新评估条件

记录会使当前选择失效的信号，例如吞吐量级、团队数量、租户隔离要求、故障预算或业务规则复杂度。避免用“以后再优化”代替可观察阈值。

## ADR 触发条件

出现下列任一情况时使用 [adr-template.md](adr-template.md)：

- 引入或替换基础设施、数据存储、通信模式或核心框架。
- 改变模块、服务、数据所有权或信任边界。
- 接受难以逆转的兼容性、可靠性、安全或成本权衡。
- 多个合理方案长期存在争议，需要保存决策背景。
