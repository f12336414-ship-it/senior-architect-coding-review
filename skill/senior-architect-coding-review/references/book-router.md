# 书籍视角路由

这些书提供问题视角，不提供凌驾于项目事实之上的命令。只选择能改变当前决策的视角。

## 路由表

| 问题 | 优先视角 | 主要检查 |
| --- | --- | --- |
| 模块或 API 难以理解 | A Philosophy of Software Design | 深模块、信息隐藏、认知负担、修改扩散 |
| 业务策略被框架控制 | Clean Architecture | 依赖方向、策略与细节、可替换边界 |
| 局部代码难读 | Clean Code | 命名、函数意图、错误处理、测试清晰度 |
| 需要系统化构造判断 | Code Complete | 防御式编程、控制流、数据和构造质量 |
| 复制、分区、事务或流处理 | Designing Data-Intensive Applications | 一致性、故障模型、数据演进、处理语义 |
| 复杂业务模型或团队语义冲突 | Domain-Driven Design | 统一语言、核心域、限界上下文、模型所有权 |
| 需要快速判断是否采用 DDD | Domain-Driven Design Distilled | 子域、上下文映射、聚合与事件的最低必要性 |

| 生产故障和依赖传播 | Release It! | 超时、隔离、背压、容量、观测和恢复 |

| 缺少测试的遗留系统 | Working Effectively with Legacy Code | 特征测试、接缝、解除依赖、增量控制 |

## 常见冲突处理

### 深模块与单一职责

按“拥有一个内聚职责”理解单一职责，不按方法或类的大小判断。一个模块可以接口很小、内部实现丰富，同时保持职责内聚。

### DRY 与允许重复

只消除同一知识的重复。外形相似但变化原因不同的代码保持独立；尚未理解变化轴时允许短期重复。

### 事务脚本与领域模型

简单 CRUD、低规则密度和短生命周期优先事务脚本。规则交互复杂、术语关键且持续演进时使用领域模型。不要为了 DDD 制造贫血对象和仪式层。

### Clean Architecture 与框架惯例

只在业务策略、外部依赖和测试边界处隔离框架。简单边缘代码可以遵循框架惯例，不要求全项目复制同心圆目录。

### 重构与重写

默认使用测试保护的小步重构。只有现有系统无法满足硬约束、替换边界清晰、迁移可渐进且双系统成本可承担时考虑重写。

### 重试与可靠性

重试不是默认可靠性机制。先确认暂时性错误、幂等性、截止时间、退避和容量影响；否则快速失败、排队或人工恢复可能更安全。

### 抽象与 YAGNI

抽象必须减少当前复杂度，或保护已观察到的变化轴。只为想象中的实现创建接口违反 YAGNI，也会形成浅模块。

## 使用限制

- 不要在输出中堆砌书名来增加权威感。
- 不要把作者建议强化为无条件规则。
- 不要一次加载十三套完整规则。
- 当项目证据与书籍启发冲突时，说明原因、风险和重新评估条件。
