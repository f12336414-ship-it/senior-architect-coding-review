# 架构级编码工作流

## 编码前

1. 确认 [design-gates.md](design-gates.md) 的 `G1` 已通过；若需求漂移，返回需求发现。
2. 写出可观察的完成行为和不应改变的行为。
3. 找到规则与数据的当前所有者，避免从便利入口直接实现。
4. 确认调用方、公开接口、数据库和事件的兼容影响。
5. 建立需求、不变量、实现和验证的最小追踪链。
6. 选择能证明行为的最小测试层级。
7. 高风险改动先通过 `G2`，设计迁移、开关、观测和回滚路径。

## 实现原则

- 把业务规则放在拥有该规则的模块，而不是控制器、序列化器或数据库适配器。
- 让接口表达业务能力和失败语义，不泄漏偶然的框架类型。
- 保持模块接口窄而完整，把可隐藏的复杂度留在模块内部。
- 对非法状态优先通过类型、构造过程或集中校验予以排除。
- 对时间、随机数、外部服务和持久化建立真实边界，但不为每个函数制造接口。
- 对跨事务或网络边界的写入明确重复、乱序、丢失和部分成功处理。
- 对资源和异步任务明确超时、取消、清理和所有权。

## 控制改动范围

- 将行为变化与纯结构重构分成可识别步骤。
- 优先沿现有架构完成垂直切片，避免顺手重写邻近模块。
- 发现更大问题时记录证据；只有它阻止当前交付或造成直接风险时扩大范围。
- 新抽象必须消除真实复杂度、重复知识或危险耦合。
- 保留临时兼容代码时，写明删除条件和观测方式。

## 遗留代码

1. 先用特征测试锁定当前可观察行为。
2. 找到可测试接缝，最小化解除依赖的改动。
3. 在新路径中建立清晰边界，再逐步迁移调用者。
4. 不把无法测试作为重写理由；先恢复反馈能力。
5. 区分需要保留的行为和可能需要修复的缺陷。

## 验证矩阵

| 改动 | 最低验证 |
| --- | --- |
| 纯函数或局部规则 | 单元测试、类型检查、静态检查 |
| 模块协作 | 集成测试和失败路径 |
| API 或事件契约 | 契约测试、旧客户端或消费者兼容 |
| 数据写入或事务 | 并发、回滚、重复执行和约束测试 |
| 数据迁移 | 生产规模演练、校验查询、备份与回退演练 |
| 外部依赖 | 超时、重试、熔断和降级测试 |
| 性能改动 | 变更前后同条件基准与资源指标 |
| 安全边界 | 未授权、越权、输入滥用和审计验证 |

## 完成检查

- 用户路径端到端成立，而非只有内部方法通过测试。
- 关键不变量在并发和失败时仍成立。
- 日志和指标能区分成功、业务拒绝、依赖失败和系统缺陷。
- 新代码没有制造双写权威、隐式全局状态或无界工作量。
- 测试验证行为，不依赖无意义的内部实现细节。
- 独立实现反证没有新增未处理的 P0/P1，交付满足 `G3`。
