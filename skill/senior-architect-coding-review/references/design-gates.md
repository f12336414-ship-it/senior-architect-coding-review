# 设计与交付门禁

## G0：需求发现启动

当业务目标、用户、范围、验收标准、规则所有者或关键失败语义不明确时进入需求发现。只要缺失内容可能改变边界、不变量、数据、安全或不可逆决策，就不得直接进入正式设计。

## G1：设计就绪

必须同时满足：

- 业务结果、利益相关者、范围和非目标明确。
- 正常、拒绝、取消、失败、恢复和人工路径足以支持当前决策。
- 核心状态、不变量、数据与授权所有者明确。
- 相关质量属性具有可验证场景、阈值或明确取舍者。
- 验收标准能够区分正确结果与局部假成功。
- 问题定义已经接受反方攻击，阻塞矛盾已解决。
- 决策性未知项已解决、隔离或由有权限者接受。

输出 `READY`、`NOT_READY` 或 `READY_WITH_ASSUMPTIONS`。最后一种只允许可逆假设，并逐条记录验证和失效信号。

## G2：实现就绪

必须同时满足：

- 至少两个真实可行方案经过对称比较；微小局部改动可以比较“保持现状”与“最小修改”。
- 推荐方案满足业务结果和不变量，且是最小充分方案。
- 数据、事务、进程、部署和信任边界清楚。
- 失败、兼容、迁移、观测和恢复语义清楚。
- P0/P1 架构攻击已关闭或升级。
- 关键需求已有实现位置与验证方法。
- High 至少获得领域负责人确认；Critical 同时获得领域负责人、相关专业审查者和风险负责人批准。

## G3：交付就绪

必须同时满足：

- 用户路径端到端成立，关键不变量在失败与并发下仍成立。
- 测试、构建、静态分析和风险专项验证已运行，或明确说明为何不能运行。
- 实现未偏离需求、ADR、兼容策略和迁移顺序。
- 关键架构规则已转化为测试、约束、静态规则或可观测信号。
- 最终实现反证没有新增未处理 P0/P1。
- 发布、暂停、回滚、前向修复和事故升级条件清楚。
- 残余风险有负责人、阈值和重新评估条件。

## 追踪链

对关键需求维护：

| 字段 | 内容 |
| --- | --- |
| Requirement | 可观察业务结果或质量属性 |
| Source | 用户、领域负责人、法规、事故、代码或指标 |
| Invariant | 必须保持的事实或承诺 |
| Decision | 满足它的架构选择 |
| Implementation | 模块、接口、约束或迁移步骤 |
| Verification | 测试、静态规则、实验或人工验证 |
| Production signal | 指标、日志、追踪或审计 |
| Owner | 规则、实现或风险负责人 |

只为影响边界、数据、安全、兼容或运行风险的需求建立正式追踪项，避免为每个局部细节制造文档。

## 残余风险

每项残余风险至少记录：

```text
风险：
影响与概率依据：
当前控制：
接受者：
成功/停止阈值：
重新评估条件：
```

代理可以发现和量化风险，不能代替授权人接受重大业务、合规、数据或生产风险。

## 身份、确认与证据

- `domain_owner`：有权定义业务规则和验收结果的团队或明确角色。
- `specialist_reviewer`：对安全、数据、可靠性、隐私或行业约束承担专业审查责任的人。
- `risk_owner`：有权限接受成本、事故、数据、合规或生产残余风险的人，不得由方案作者默认兼任。

有效批准至少记录身份、角色、批准范围、UTC 时间和不可变证据链接。口头确认只有被记录到 Issue、PR、ticket、邮件归档或批准系统后才构成证据。批准仅覆盖记录的版本、范围和风险；方案、风险或证据发生实质变化时重新批准。

Unknown 使用 `open -> resolved | accepted | obsolete` 生命周期。blocking unknown 不能以 `accepted` 关闭，除非有权限的风险负责人记录接受范围和重新评估条件。
