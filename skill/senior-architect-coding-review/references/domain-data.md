# 领域、事务与数据设计

## 选择模型复杂度

- 简单数据录入、报表或低变化 CRUD：优先事务脚本和清晰服务边界。
- 存在复杂不变量、状态机、策略和长期演进：建立领域模型和统一语言。
- 多团队或语义冲突：识别子域、限界上下文和上下文映射。
- 不因使用微服务而自动使用 DDD，也不因使用单体而拒绝领域边界。

模型应表达真实业务决策。只有数据字段和 getter/setter 的对象不是领域模型。

## 所有权与不变量

- 为每项业务规则和权威数据指定单一所有者。
- 将必须原子成立的不变量放在同一事务或一致性边界中。
- 聚合保持足够小，只包含必须强一致的状态。
- 跨聚合或跨服务规则显式选择最终一致、补偿或协调机制。
- 不让多个服务直接写入同一逻辑数据而没有明确协议。

## 事务与并发

明确以下语义：

- 隔离级别是否足以阻止丢失更新、写偏差或幻读？
- 使用乐观锁、悲观锁、唯一约束还是幂等键保护竞争？
- 重试是否会重复副作用？提交结果未知时如何恢复？
- 数据库写入和消息发布如何避免双写不一致？
- 批处理失败时如何续跑、校验和隔离坏数据？

优先使用数据库约束保护不可违反的数据事实，应用层负责业务语义和友好错误。

## 事件与消息

- 事件表示已发生的领域事实，不是隐藏的远程过程调用。
- 明确 schema、版本、分区键、顺序、重复和保留策略。
- 消费者默认面对至少一次投递，使用幂等处理或去重。
- 不承诺消息与数据库原子性，除非机制能够证明；必要时使用 outbox/inbox。
- 事件最终一致必须定义可接受延迟、用户体验和补偿流程。
- 避免事件链形成不可追踪的业务控制流。

## 数据模型与演进

- 根据访问模式、约束和生命周期选择模型，不按潮流选择存储。
- 区分事实来源、派生视图、缓存和搜索索引。
- schema 变化默认需要新旧代码并存窗口。
- 明确保留、删除、审计、隐私和数据地域要求。
- 对大规模数据操作先估算时间、锁、日志、复制和存储影响。

## 判断完成

能够回答：谁可以修改什么数据、什么必须立即一致、什么允许延迟、失败后如何检测和修复、模型如何兼容演进。
