# 兼容演进与迁移

## 总体策略

默认采用 `expand -> migrate -> contract`：

1. **Expand**：先增加向后兼容的新能力，新旧版本可以共存。
2. **Migrate**：迁移流量或数据，持续校验并处理差异。
3. **Contract**：确认旧路径无使用者后，再删除兼容代码和旧结构。

不要依赖所有实例、客户端和消费者同时升级。

## 数据库迁移

- schema 变更和数据回填分开部署。
- 大表操作评估锁、事务日志、复制延迟、磁盘和执行时间。
- 先增加可空字段或新表，再部署双读/双写或回填，最后收紧约束。
- 双写必须定义权威来源、失败处理、差异检测和结束条件。
- 回填任务可暂停、续跑、幂等，并记录进度和错误对象。
- 使用计数、校验和、抽样或业务不变量验证迁移结果。
- 备份只有经过恢复演练才构成回滚证据。

## API 和协议

- 区分增加字段、改变含义、改变默认值和删除字段；后者通常是破坏性变化。
- 读取者容忍未知字段，写入者不要假设所有消费者已升级。
- 版本化无法替代弃用期限、使用监测和消费者沟通。
- 错误码、排序、分页、幂等和时间语义也属于契约。
- 对公开接口使用消费者或契约测试验证旧行为。

## 事件演进

- 事件 schema 只增加兼容字段，改变语义时使用新事件或明确版本。
- 确认历史事件能否被新消费者读取和重放。
- 在删除字段或主题前用消费指标证明没有活跃消费者。
- 迁移期间监测新旧消费者处理差异、积压和死信。

## 发布与回滚

- 指定代码、配置、schema、任务和流量切换的精确顺序。
- 使用开关时定义默认值、权限、监测、过期时间和删除责任。
- 回滚前检查新代码是否写入旧代码无法理解的数据。
- 对不可逆数据变化准备前向修复，不虚构一键回滚。
- 定义继续、暂停、回滚和升级事故的客观阈值。

## 完成条件

- 新旧版本已在真实路径并存验证。
- 数据差异和兼容指标达到约定阈值。
- 旧路径使用量为零或有明确例外。
- 临时代码、开关、影子表和指标已有删除计划与所有者。
