# 项目事实扫描

## 目标

用最少读取建立足够可靠的事实基线。不要把仓库规模等同于需要阅读的范围。

## 证据优先级

1. 可重复的测试、构建、运行结果和生产指标。
2. 当前代码、数据库迁移、API/事件模式和部署配置。
3. 已接受的 ADR、架构文档和项目规则。
4. Issue、PR、提交历史和事故复盘。
5. 命名、目录结构和注释所表达的意图。
6. 模型常识或未经验证的推断。

证据冲突时优先相信当前可执行行为，并记录文档漂移。

## 最小扫描顺序

1. 定位全局和目录级 `AGENTS.md`，确认适用范围。
2. 读取 README、构建配置、依赖清单和测试入口。
3. 定位用户请求对应的入口、核心逻辑、边界适配器和测试。
4. 沿调用链和数据流双向追踪一层，识别调用者与被调用者。
5. 检查相关 schema、迁移、API、事件、权限和配置。
6. 检查当前 diff 和邻近提交，识别正在进行的设计变化。
7. 只在关键问题仍未回答时扩大搜索。

## 必须回答的问题

- 业务能力由哪个模块拥有？
- 数据的权威来源和写入者是谁？
- 公开契约和兼容承诺是什么？
- 事务、进程、网络和信任边界在哪里？
- 当前测试实际保护哪些行为？
- 系统如何部署、观测和回滚？
- 现有设计是有意决策、历史妥协，还是偶然结构？

## 事实记录

对重要判断使用以下标签：

- `FACT`：有文件、行号、命令输出或文档支持。
- `INFERENCE`：由多个事实推导，说明推理链。
- `UNKNOWN`：仓库中无法确认，说明可能影响。
- `ASSUMPTION`：为继续工作采用的可逆假设。

关键证据同时记录来源位置、获得时间、适用环境和它实际支持的结论。不要让单元测试证明生产容量，也不要让文档意图覆盖当前可执行行为。

## 任务证据图

只为当前决策建立下列关系：

```text
需求/验收
  -> 入口与调用者
  -> 规则和状态变化
  -> 持久化或外部边界
  -> 测试与运行信号
  -> 发布、恢复与负责人
```

发现证据冲突时，不要静默选择对方案最有利的一项。记录冲突，确认是环境差异、文档漂移、测试缺陷还是需求本身尚未决策。

## 停止读取条件

当以下条件同时满足时停止扩大上下文：

- 能描述改动入口到持久化或外部边界的路径。
- 能指出关键不变量、所有者和失败语义。
- 能确定验证命令和兼容影响。
- 能把关键需求追踪到实现位置和验证证据。
- 新读取的文件不再改变设计判断。

不要先生成全仓库架构说明。只为当前决策建立足够上下文。
