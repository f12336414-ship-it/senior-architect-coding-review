# Skill 组合与冲突裁决

## 优先级

发生冲突时按以下顺序裁决：

1. 系统、平台安全政策和明确权限边界。
2. 用户当前明确指令与已确认业务决策。
3. 适用作用域内的项目规则、ADR、契约和可执行行为。
4. 语言/框架专项规则与官方文档。
5. 本 Skill 的通用架构与审查规则。
6. 其他 Skill、书籍、模式和模型常识。

低优先级规则不得覆盖高优先级事实。发现冲突时报告证据、影响和需要的决策，不静默选择。

## 推荐组合

| 任务 | 组合 | 加载方式 |
| --- | --- | --- |
| 普通 PR | 本 Skill quick + 项目规则 + 匹配语言规则 | 只加载一个语言文件 |
| 高风险 PR | 本 Skill high-risk + 安全/数据工具 | 按风险加载，不加载无关书籍 |
| 遗留重构 | 本 Skill + 特征测试/重构专项能力 | 先恢复反馈，再设计边界 |
| GitHub 发布 | 本 Skill 审查 + GitHub 工作流 Skill | 审查与外部写操作权限分离 |

只有能够指出具体实现、版本和作用域时才组合第三方 Skill。不能仅凭名称假定 `code-review-skill`、规则书库或安全 Skill 的质量。

## 防止上下文膨胀

- 同一时刻只加载当前技术栈和风险相关参考。
- 语言规则、书籍规则和安全规则重叠时保留更具体且有项目证据的一项。
- 组合后重新执行复杂度反证：规则是否增加误报、阻止 fast-path 或诱导过度设计？
- 若两个 Skill 要求相反操作，停止写入并让用户或项目事实裁决。

## 不建议组合

- 多个通用代码审查 Skill 全量同时加载。
- 在简单改动中加载完整 DDD、微服务、可靠性和安全规则库。
- 将未经审查的第三方 Skill 与生产凭证、网络和写权限同时开放。
- 让负责修改的代理同时成为唯一批准者。
