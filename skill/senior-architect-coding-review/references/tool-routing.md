# 工具与架构适应度路由

## 原则

先发现仓库已经使用的构建、测试、分析和部署工具，再选择验证命令。不要未经用户授权安装工具或把建议工具描述成已运行证据。

工具用于证明可检查属性，不代替需求、领域和组织判断。

## 项目能力发现

必要时运行：

```bash
python scripts/detect_project_capabilities.py .
```

同时读取构建清单、CI、测试配置和 `AGENTS.md`。脚本只提供候选能力，实际命令以项目文档和配置为准。

## 验证路由

| 风险 | 优先证据 |
| --- | --- |
| 局部业务规则 | 单元测试、类型检查、静态检查 |
| 模块边界 | 依赖规则、循环检查、架构测试 |
| API/事件 | schema diff、契约测试、旧消费者测试 |
| 数据写入 | 数据库约束、并发、回滚和幂等测试 |
| 数据迁移 | dry-run、锁与容量评估、校验、恢复演练 |
| 外部依赖 | 超时、重试、取消、隔离和降级测试 |
| 性能 | 同条件基准、尾延迟、资源和容量指标 |
| 安全 | SAST、依赖与秘密扫描、越权和滥用测试 |
| 发布 | 兼容窗口、开关、金丝雀、停止阈值和回滚 |

## 常见架构工具

- Java：ArchUnit 检查包、层、切片和循环依赖。
- .NET：ArchUnitNET 检查程序集与类型依赖。
- Python：Import Linter 约束模块导入方向。
- JavaScript/TypeScript：dependency-cruiser 或 ArchUnitTS 检查依赖规则。
- 多语言：Semgrep 执行项目规则与安全模式；CodeQL 执行跨过程安全和质量查询。

只有仓库已采用、用户批准引入，或风险确实需要时才配置新工具。优先把长期架构承诺转化为项目原生测试和 CI 规则。

## 证据强度

从弱到强：

1. 命名、目录和注释表达的意图。
2. 静态代码阅读与推理。
3. 静态分析、类型、schema 和架构规则。
4. 可重复测试、构建和实验。
5. 真实规模演练、生产指标与恢复证据。

结论强度不得超过证据强度。性能、可用性、恢复和安全声明通常需要专项验证，不能仅凭代码阅读关闭。
